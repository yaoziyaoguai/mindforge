# ADR 0004: Deprecate Megadoc Wiki and Adopt Topic Presenter

## Status
Accepted (partially implemented as of 2026-06-04)

**Implementation status:**
- `llm_rebuild_wiki`: fully disabled (raises `WikiError`) — all entry points frozen
- `rebuild_main_wiki`: fully disabled (raises `WikiError`) — all entry points frozen
- CLI `wiki rebuild`: hidden, prints deprecation notice, exits 0
- Web `POST /api/wiki/rebuild`: returns 410 Gone
- `approval_service` auto-rebuild: removed, returns deprecation notice
- `TopicPresenter`: implemented with real summary extraction, approval boundary enforcement
- `GET /api/topics`: implemented with stable `TopicListResponse` schema
- `GET /api/topics/{topic_name}`: implemented with stable `TopicViewResponse` schema
- Frontend reconstruction: **not yet started** — blocked on API contract lock-in (now complete)
- `Main-Wiki.md` read endpoints: still available for legacy read access; new content not written

## Context
MindForge's current Knowledge Experience relies on a "Megadoc Wiki" (`30-Wiki/Main-Wiki.md`). This file is generated by concatenating `human_approved` cards or, more dangerously, by using an LLM to synthesize sections based on those cards (`llm_rebuild_wiki`).

This approach has revealed severe architectural and product flaws:
1.  **Approval Boundary Violation**: The `llm_rebuild_wiki` function generates new text (summaries, overviews) that bypasses the `ai_draft` -> `human_approved` pipeline. It presents unverified LLM hallucinations as "Main Wiki" content, violating the core local-first, approval-first philosophy.
2.  **Scalability & UX Disaster**: A single Markdown file containing hundreds of cards is unreadable. The frontend (`WikiPage.tsx`) simply renders an accordion list, offering no structural hierarchy or semantic navigation.
3.  **Anemic Knowledge Model**: Cards are treated as flat text containers. Relationships ("Related Sections") are calculated using a mathematical Jaccard overlap of shared cards, rather than explicit semantic links (e.g., "supports", "contradicts").

## Decision
We will completely reconstruct the Knowledge and Wiki subsystems:

1.  **Deprecate `Main-Wiki.md`**: We will no longer write to a monolithic Wiki file. Existing files will be ignored (left as legacy artifacts).
2.  **Enforce Strict Approval Boundaries**: `llm_rebuild_wiki` is now disabled (raises `WikiError`). `rebuild_main_wiki` is also disabled. All LLM-generated synthesis *must* enter the system as a distinct card with `status: ai_draft` and `knowledge_type: summary`. It cannot be displayed in the core knowledge views until a human explicitly approves it.
3.  **Adopt MVC/Presenter Architecture**: The Wiki will transform from a persisted file into a **Runtime View**.
    *   **Backend (`TopicPresenter`)**: Will query approved cards, group them by Topic/Track, resolve explicit semantic relations, and serve JSON.
    *   **Frontend**: Will render a Three-Pane "Topic Browser" instead of a flat list.
4.  **Introduce Knowledge Model v2**: Cards will be enriched with `knowledge_type` (concept, claim, insight, etc.) and explicit `relations` (supports, expands, etc.) in their YAML frontmatter.

## Consequences
*   **Positive**: We restore the absolute integrity of the `human_approved` boundary. The knowledge base becomes a true semantic network rather than a text dump. Frontend UX is vastly improved.
*   **Negative**: The frontend requires significant rework. We must implement a compatibility layer for old cards that lack explicit types and relations.
*   **Migration**: No data migration scripts are required. The changes to the YAML schema are strictly additive, and parsers will use safe fallbacks (`knowledge_type: concept`) for legacy cards.
