# MindForge v0.13 Stage 0 — Non-Sensitive Dogfooding Readiness

> **Status:** Stage 0 (planning + scaffolding only). Stage 0 ships
> **no** new runtime, **no** new commands, **no** real provider, **no**
> private data. It pins the safety contract, fixtures, and readiness
> checklist that any later v0.13 stage must respect before turning real.
>
> **Relationship to v0.12:** v0.12 closed the declarative custom-strategy
> preview path (loading / discovery / preview packet, all review-only,
> never executable). v0.13 is the next outward step — getting MindForge
> ready for *non-sensitive* end-to-end dogfooding without ever silently
> reading user secrets or writing user vaults. Stage 0 makes that
> readiness explicit.

---

## 1. Why a "Stage 0" at all

Industry pattern observation (offline judgement, drawing on prior
knowledge of OpenAI Agents SDK guardrails docs, LangGraph
human-in-the-loop guides, Dify Knowledge Pipeline node UX, Obsidian's
local-first stance, and the Readwise / Logseq / Tana / Cubox second-brain
families):

- Every credible local-first AI tool that survives contact with real users
  pins its safety contract **before** it pins its features.
- Tools that skip that step end up retrofitting kill-switches under
  pressure (auto-approve toggles, hidden cloud sync, accidental vault
  rewrites, env-leak incidents).
- MindForge's whole identity is fake-first + ai_draft-only + explicit
  approval. Dogfooding is the moment that identity gets stress-tested.

Stage 0 therefore exists to write down — in code-checkable form — what
"non-sensitive dogfooding" means here, so that Stage 1+ implementers
(human or AI) cannot drift.

---

## 2. Local-first dogfooding safety contract

A v0.13 dogfooding session is *non-sensitive* iff **all** of the
following hold:

1. **Inputs** come from `examples/` (synthetic fixtures) or from a path
   the user explicitly passes on the command line. No implicit scans of
   `~`, `~/Documents`, `~/Library`, `~/Obsidian`, `~/Downloads`, etc.
2. **No `.env` is read**, ever, in the dogfooding flow. Real-provider
   opt-in remains a separately-authorized v0.13+ surface.
3. **No real LLM provider is contacted**. The fake provider chain is
   the only allowed runtime.
4. **No real Cubox API call is made**. Cubox dry-run path stays the only
   touched code, and only against synthetic JSON fixtures.
5. **No write occurs into a real Obsidian vault**. `examples/demo-vault/`
   is the only legal write target; even there, writes must be opt-in
   via existing CLI flags.
6. **No human_approved card is generated by automation**. Approval
   remains an explicit user gesture, the same as today.
7. **No custom strategy is executed.** Custom strategies are still
   preview-only per v0.12 capability matrix.
8. **No arbitrary Python plugin / shell strategy / RAG / embedding /
   semantic merge** is introduced.
9. **No tag, no force push, no remote modification** beyond reviewed
   pushes.
10. **Every dogfooding session must be a dry-run by default**. Mutation
    requires an explicit, separate flag the user types every time.

Violation of any of the above means the work is **not** v0.13 Stage 1+;
it is a different milestone and needs its own RFC.

## 4. Human decision gate map

Borrowing the "human-in-the-loop interrupt" framing from LangGraph and
the "human approval" guardrail framing from OpenAI Agents SDK, v0.13
dogfooding has these **mandatory** human decision gates. None can be
auto-resolved.

| Gate | Where today | Stage 1+ must keep |
| --- | --- | --- |
| Approve an `ai_draft` knowledge card | `mindforge approve` flow | yes — approval stays explicit |
| Confirm Obsidian write | existing obsidian CLI flags | yes — write stays opt-in |
| Activate real LLM provider | (does not exist yet, planned v0.13+) | must be a separate, audited opt-in |
| Activate real Cubox API | (does not exist yet) | must be a separate, audited opt-in |
| Promote a custom strategy from `preview_only` to executable | (does not exist yet) | must be a separate RFC, not a CLI flag |

Stage 0's job is to write this map down so Stage 1+ does not bypass any
gate by accident.

## 5. Local-first privacy contract

> **v0.13 Stage 1 update**: 这一节的 canonical 版本已迁移至
> [`LOCAL_FIRST_PRIVACY_CONTRACT.md`](./LOCAL_FIRST_PRIVACY_CONTRACT.md);
> 该文档新增了 fake-default + real-opt-in 的细化条款 (例如
> `--allow-real` flag 语义、Real ≠ Approved 硬隔离、Human Decision Gate
> Map、testable invariants)。本节保留为历史背景, 与 canonical 文档
> 出现差异时以 canonical 文档为准。

Inspired by Obsidian's "your notes live on your disk in plain markdown"
stance and the Readwise / Cubox capture-side products' opt-in sync
model. MindForge promises:

- **No background indexing of the user's home directory.**
- **No implicit scanning of any Obsidian vault.** Vault paths must be
  explicit CLI inputs.
- **No cloud sync of any kind by default.**
- **No telemetry by default.** (Telemetry plans, if any, must be
  separately RFC'd and opt-in.)
- **No silent network calls.** Every outbound network call must trace
  back to a CLI flag the user typed in this session.
- **No silent writes.** Mutation of vault, cards, or workspace requires
  either dry-run mode (default safe) or an explicit mutation flag.

Stage 0 fixes this contract in writing so Stage 1+ has something to
break against, not something to discover.

## 6. Capability matrix delta vs v0.12

The v0.12 capability matrix already encodes "custom preview is
review-only, never executable." v0.13 Stage 0 adds nothing executable;
the capability matrix delta is therefore **empty by design**:

| Lifecycle stage | v0.12 status | v0.13 Stage 0 delta |
| --- | --- | --- |
| Definition / Loading / Discovery | done | unchanged |
| Build / activate (custom) | refused | unchanged |
| Preview packet | review-only | unchanged |
| Execution (custom) | future | still future |
| Approval | explicit only | unchanged |
| Writer | post-approval only | unchanged |

Stage 0 only adds **review-readable artifacts** (this doc + a synthetic
fixture + assertion tests). Nothing in this delta could leak data,
write a vault, or call an LLM.

## 9. Stop conditions for Stage 0

Stage 0 ends — and **must** end — when:

- This doc + the synthetic fixture + the readiness assertion tests are
  green;
- No production module under `src/mindforge/` was modified;
- No new dependency was added;
- No tag was created;
- No commit was force-pushed;
- The full test suite remains green.

Crossing any of those lines means the work is no longer Stage 0; it is
Stage 1+ and needs its own approval.

## 10. Future architecture proposal placeholder

During Stage 0 research I considered whether the preview packet itself
should grow into a generic "ReviewableArtifact" boundary object shared
by ai_draft / approval / preview / dry-run-report paths. **I am not
proposing that change now.** It is recorded here so a later RFC can
pick it up:

> **Proposal sketch (not for v0.13 Stage 0 implementation).** A
> `ReviewableArtifact` protocol could give CLI / future UI a single
> consumption surface for "anything the user is being asked to review
> before committing." Today preview packet, ai_draft card, and
> dry-run-report each have their own shape. Unifying them is *not*
> required for safety; it is a UX cohesion question. Worth an RFC only
> if v0.13 Stage 1+ surfaces concrete duplication pain. Otherwise this
> stays deliberately unimplemented.

That sketch lives only in docs and explicitly does **not** authorize
any refactor.
