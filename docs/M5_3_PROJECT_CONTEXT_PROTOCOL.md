# M5.3 · Project Context Protocol（v0.2.2）

> 本文是 **M5.3 Better project context** 的产品 + 协议约定。
> 上一版：[`M4_RECALL_REVIEW_PROTOCOL.md`](./M4_RECALL_REVIEW_PROTOCOL.md) §6（v0.2.0）+ M4.1 增量（v0.2.1）。
> 本版只增不破：v0.2.0 / v0.2.1 既有 CLI 调用与 markdown / json 字段全部保留。

---

## 0. 定位（先讲清楚不是什么）

`mindforge project context` **不是**：
- ❌ 不是 RAG / embedding / 向量检索
- ❌ 不是 LLM 总结（任何阶段都不调 LLM）
- ❌ 不是自动生成新知识
- ❌ 不是浏览器插件 / Obsidian 插件
- ❌ 不读取 `.env`、`runs/`、`state.json` 作为知识来源
- ❌ 不读取 `00-Inbox/` 任何 raw source

`mindforge project context` **是**：
- ✅ 从 **已确认的 Knowledge Cards** 与 **项目 profile 文件** 拼装一份
  **安全、结构化、可复制**给编程助手的上下文包
- ✅ 输出每一段都能追溯到来源（project profile 还是 cards）
- ✅ 默认仅 `human_approved` 卡片；`--include-drafts` 才包含 `ai_draft`
- ✅ 适配 `claude-code` / `copilot` / `codex` / `generic` 四种目标

---

## 1. 数据源（混合：项目级 + 卡片级）

### 1.1 项目级（优先级最高）

读取路径：`<vault.root>/<vault.projects_dir>/<project_name>.md`
（默认 `projects_dir = "30-Projects"`，与 vault_template 一致）

frontmatter 可选字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `project` | string | 显示名（缺省时回退到 `project_name`） |
| `description` | string | 一句话描述 |
| `default_target` | string | `claude-code` / `copilot` / `codex` / `generic` |
| `principles` | list[string] | 项目级原则（**优先级最高**） |
| `known_risks` | list[string] | 已知风险 / 反模式 |
| `preferred_workflow` | list[string] | 推荐工作流 |

正文（frontmatter 之后的内容）**不被读取**。这避免了"项目笔记里随手写
了 secret / 草稿"被误读进 context pack。

### 1.2 卡片级（补充，永不覆盖）

匹配条件：`projects: [<project_name>]` ∈ frontmatter；默认 `status=human_approved`。

新增可选字段（backwards-compatible，缺失即空）：

| 字段 | 类型 | 用途 |
|---|---|---|
| `principles` | list[string] | 卡片中沉淀的方法 / 原则 |
| `known_risks` | list[string] | 卡片中提到的反模式 / 风险 |

> 注：v0.2.0 已有的 `Action Items` / `Reusable Prompts / Principles`
> 段在卡片**正文**里抽取（仍按 §M4 §6.2），`principles` 这个 frontmatter
> 字段是**结构化**的补充，二者并存、不替代。

### 1.3 降级策略

- 项目 profile 文件**不存在**时：**不**报错；仅从卡片聚合，并在输出顶部
  显式声明 `No project profile found at <path>; using Knowledge Cards only.`
- 项目 profile 文件**存在但 frontmatter 无法解析**：归类为
  `project_profile_found=false` + `error_message`，仍走 cards-only 降级。
- 项目 profile 中所有字段都为空 / 缺失：每段输出仍要展示，文案为
  `No project-level <field> configured.`，**绝不输出空标题**。

## 2. CLI 协议

### 2.1 新增参数

```
mindforge project context <PROJECT> \
  [--target {claude-code|copilot|codex|generic}] \
  [--format {markdown|json}] \
  [--output FILE | -o FILE] \
  [--limit N] \
  [--include-drafts] \
  [--include-actions / --no-actions] \
  [--include-review-due / --no-review-due] \
  [--include-next-step-prompt / --no-next-step-prompt] \
  [--no-prompts]
```

新增的只有 `--target`；其它全部是 v0.2.0/v0.2.1 既有，行为不变。

### 2.2 `--target` 解析顺序

1. 命令行 `--target` 显式指定 → 用它
2. 否则若项目 profile 含 `default_target` → 用它
3. 否则 → `generic`

不合法的 `--target` 值 → exit 2，错误打印到 stderr。

### 2.3 退出码

| 场景 | exit |
|---|---|
| 正常 | 0 |
| `--output` 父目录不存在 | 2 |
| `--target` 不在四枚举内 | 2 |
| `--format` 不在 `markdown|json` | 2 |

## 3. Markdown 输出结构（v0.2.2）

> 所有段都**始终**渲染。无内容时显示降级文案，**绝不**出现空标题。
> 段落顺序固定，便于人类与脚本同时消费。

```
# Project Context · <project_name>

> Generated by mindforge project context at <iso>
> target: <target>
> 仅含 status=human_approved 的卡片摘要（除非 --include-drafts）
> 字段范围：见 docs/M5_3_PROJECT_CONTEXT_PROTOCOL.md

## Source Notice
- project_profile_found: true|false
- project_profile_path: <vault-relative path or "(not found)">
- card_count: <int>

## Project Profile
- description: <str | "No description configured.">
- preferred_workflow:
  - ...   (or "No preferred_workflow configured.")

## Project-Level Principles
- ...   (or "No project-level principles configured.")

## Card-Level Principles (supplementary)
- from [card-id]: ...   (or "No card-level principles found.")

## Project-Level Known Risks
- ...   (or "No project-level known_risks configured.")

## Card-Level Known Risks (supplementary)
- from [card-id]: ...   (or "No card-level known_risks found.")

## Related Learning Tracks
- <track> (<n> cards)   (or "(none)")

## Knowledge Cards (sorted by review_after asc, then value_score desc)
### [card-id] Title
- track / status / value_score
- source / source_url
- reviewed_at / review_after
- path
#### Source Excerpt   (if present)
#### AI Summary       (if present)
#### Reusable Prompts / Principles   (if present and --include-prompts)

## Action Items                       (if --include-actions)
## Review Due                         (if --include-review-due)

## Suggested Prompt for <target>      (always; target-specific fixed template)
```text

```

## Excluded Content (safety guarantee)
- raw source / 00-Inbox 文件正文
- AI Inference (low confidence)
- Human Note 段
- secrets / api_key / Authorization headers
- runs/ jsonl 与 state.json
- prompt 全文与 LLM completion

### 3.1 与 v0.2.0/v0.2.1 的兼容性

- v0.2.0 已有的 `Safe Context Hint for LLM Tools` 与 v0.2.1 的
  `Recommended Next-step Prompt for LLM Tools` 段被 §3 中的
  `Suggested Prompt for <target>` **取代**（功能等价但更结构化）。
- 既有 `--include-next-step-prompt` 标志现在控制 **Suggested Prompt** 段
  是否输出（默认仍 ON）。
- 旧 markdown 输出中 `## Knowledge Cards` / `## Action Items` /
  `## Review Due` / `## Related Learning Tracks` 字面段名保持不变。

## 4. Suggested Prompt 模板（按 target，固定文本，**不调 LLM**）

每个 target 模板由两段组成：
1. **角色 + 上下文边界**（所有 target 共享前缀）
2. **target-specific 工作风格指令**（只取决于 target 与项目 profile 中
   的 `principles` / `preferred_workflow` 是否存在；不读卡片正文）

### 4.1 共享前缀

```
你是我在 <project_name> 项目上的协作助手（target=<target>）。
mindforge 已经把已审核的项目上下文打包给你（project profile + 卡片
摘要 + 行动项 + 风险 + 复习清单）。请严格按以下规则工作：

1. 不要捏造没有出现在上下文中的事实；
2. 你的产出要明确区分「来自上下文的事实」与「你的推断」；
3. 不要试图读取或修改 .env / runs / state.json；
4. 涉及代码改动时，先列文件路径、再给 diff 或最小补丁；
5. 如果项目 profile 列出了 principles / known_risks，请优先遵守。
```

### 4.2 target-specific 段

| target | 风格指令（固定） |
|---|---|
| `claude-code` | 先看真实事件链路（日志 / checkpoint / runtime_observer / state / 真实 events）再决定改不改代码；不要只根据 UI 表象打补丁；产出 3 步建议：next-action / verification / risk。 |
| `copilot` | 先 plan（列出受影响文件 + 测试范围）再小步实现；每个改动后必须给出验证命令（pytest / ruff / smoke）；遇到不确定的设计点，**先**给 2–3 个选项与 trade-off，让我选，再实施。 |
| `codex` | 仅提交"高质量补丁 + 测试"；不引入未列出的依赖；不做超出请求范围的重构；如发现 root cause 与请求不一致，先停下来汇报再继续。 |
| `generic` | 通用安全开发版本：plan → patch → test → report；任何破坏性操作（删文件、改 schema、改 lockfile）必须先确认。 |

### 4.3 与项目 profile 的合并

如果项目 profile 中有 `principles` / `preferred_workflow`，模板尾部追加：

```
项目级 principles：
- ...
项目级 preferred_workflow：
- ...
```

> 这是**字符串拼接**，不是 LLM 生成。

## 5. JSON 输出 schema（v0.2.2）

`--format json` 输出顶层字段：

```jsonc
{
  "version": 2,                       // v0.2.2 升到 2（v0.2.0/v0.2.1 是 1）
  "project": "<project_name>",
  "generated_at": "<iso>",
  "target": "<target>",
  "include_drafts": false,
  "include_prompts": true,
  "include_actions": true,
  "include_review_due": true,
  "include_next_step_prompt": true,
  "count": <int>,

  "project_profile_found": true|false,
  "project_profile_path": "<vault-relative or null>",

  "project_level_principles": ["..."],
  "card_level_principles": [
    {"card_id": "...", "items": ["..."]}
  ],
  "project_level_known_risks": ["..."],
  "card_level_known_risks": [
    {"card_id": "...", "items": ["..."]}
  ],
  "project_description": "<str|null>",
  "preferred_workflow": ["..."],

  "items": [ /* 与 v0.2.1 相同的卡片结构 */ ],

  "suggested_prompt": "<完整模板字符串>",

  "excluded_content": [
    "raw source (00-Inbox/**)",
    "AI Inference (low confidence)",
    "Human Note section",
    "secrets / api_key / Authorization headers",
    "runs jsonl / state.json",
    "LLM prompt full text / completion text"
  ]
}
```

> v0.2.2 的 JSON `version: 2` 表示**有新字段**，但 v0.2.0/v0.2.1 已有
> 字段的语义与名称都不变；下游消费者用 `if "project_level_principles"
> in payload` 这种 forward-compatible 写法即可。

## 6. RunLogger 字段扩展

`run_logger._ALLOWED_FIELDS` 新增：

- `target` — 实际生效的 target 字符串（claude-code / copilot / codex / generic）
- `project_profile_found` — bool

`project_context_emitted` 事件示例：

```json
{
  "event": "project_context_emitted",
  "project_name": "my-first-agent",
  "count": 5,
  "output_format": "markdown",
  "target": "claude-code",
  "project_profile_found": true
}
```

> **不**记录 keyword 原文、profile 内容、principles 文本。

## 7. 安全反模式（M5.3 永远不允许的事）

| 反模式 | 为什么禁止 |
|---|---|
| 把项目 profile **正文**当 context 输出 | 个人项目笔记最容易写 secret / 草稿 |
| 在 suggested prompt 里调 LLM 生成模板 | 任何 LLM 调用都会产生不可复盘的随机串 |
| 用项目名做路径拼接时不防 `..` / 绝对路径 | 防止 `mindforge project context ../../etc/passwd` |
| 用环境变量 / `.env` 决定 default_target | default_target 必须是 vault 内的"事实" |
| 把卡片正文 hash 放进 telemetry | 内容指纹仍可能与原文配对泄露 |
| `--target` 之外做"智能 target 推断" | 任何"猜你想要"都是不可解释的副作用 |

## 8. 项目 profile 示例

`30-Projects/my-first-agent.md`：

```markdown
---
project: my-first-agent
description: "个人 Agent Runtime 实验项目"
default_target: claude-code
principles:
  - "先看日志、checkpoint、runtime_observer、session、状态字段和真实事件链路，再决定是否改代码。"
  - "关键代码必须有中文学习型注释/docstring，解释架构边界和为什么这么做。"
known_risks:
  - "不要只根据 UI 表象打补丁。"
  - "不要让模型自动批准 human_approved。"
preferred_workflow:
  - "先做只读诊断。"
  - "再做最小补丁。"
  - "最后跑测试和 smoke。"
---

# my-first-agent

（这里的正文 mindforge **不会读取**；可以放任何个人笔记。）
```

## 9. 验收（v0.2.2 落地标准）

1. 项目 profile 存在时，markdown 与 json 输出都把 project-level
   principles / known_risks / description / preferred_workflow 优先展示。
2. 项目 profile 不存在时，不报错，输出顶部声明降级。
3. 卡片中的 frontmatter `principles` / `known_risks` 作为 supplementary
   分段展示，**不**与项目级合并去重（来源透明优先于 dedup）。
4. markdown 输出**不**出现空标题；缺数据用降级文案。
5. JSON `version: 2`，新字段稳定，旧字段不变。
6. `suggested_prompt` 内容随 `--target` 变化；未调 LLM。
7. `--target invalid` exit 2；`--output` 父目录不存在 exit 2。
8. `runs jsonl` 仅含白名单字段；测试断言不含 secret / 卡片正文 / profile 正文。
9. v0.2.0 / v0.2.1 既有 17 个 project context 测试 + recall / review 测试
   全部继续通过。
